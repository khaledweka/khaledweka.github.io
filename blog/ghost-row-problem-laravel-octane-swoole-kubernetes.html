Laravel Octane + Swoole in Kubernetes

The Setup — Octane + Swoole in Production

Laravel Octane with Swoole is a massive performance upgrade. Instead of bootstrapping your entire application on every HTTP request (like PHP-FPM does), Swoole keeps your application in memory and reuses long-lived workers to handle requests. The result: dramatically lower latency, higher throughput, and significantly less overhead.

But this architecture introduces a class of bugs that doesn't exist in traditional PHP. The most dangerous one involves database connection persistence.

In PHP-FPM, every request gets a fresh process. Fresh memory, fresh connections, fresh state. When the request ends, everything is destroyed. It's inefficient, but it's safe.

In Swoole, workers are long-lived. Database connections persist across hundreds of requests. And that's where ghost rows come from.

What Is a Ghost Row?

A ghost row is an INSERT that your application believes succeeded — it gets a valid auto-increment ID, builds a successful response, returns HTTP 201 — but the row never actually persists to disk. Moments later, it vanishes from the database. The auto-increment counter moves forward (InnoDB never rolls that back), so you're left with a valid-looking ID that points to nothing.

The Danger

Your API returns { "id": 247, "status": "created" } to the client. The client stores this ID, builds UI around it, maybe triggers downstream processes. But SELECT * FROM claims WHERE id = 247 returns empty. The row never existed — or rather, it existed for a few milliseconds inside an uncommitted transaction.

Why Swoole Workers Carry Dirty State

The root cause is a phantom transaction — an uncommitted BEGIN left behind by a previous request on the same worker. This happens when:

A request starts a transaction but hits an unhandled exception, a timeout, or a connection hiccup before COMMIT or ROLLBACK executes. In PHP-FPM, the process dies and MySQL auto-rolls back. In Swoole, the worker survives. The connection survives. The uncommitted BEGIN survives.

The next request hits the same worker, inherits the same connection, and calls DB::transaction(). Laravel sees an existing connection with an open transaction. Instead of issuing a new BEGIN, it creates a savepoint — a nested checkpoint inside the existing transaction.

The Kill Sequence — Step by Step

Request A → Worker #1 (leaves dirty state)
1
BEGIN — transaction starts
↓
2
INSERT / UPDATE operations execute
↓
3
Exception / timeout occurs — COMMIT never fires
↓
4
Worker returns to pool with open transaction on connection
Request B → Same Worker #1 (your claim creation)
1
DB::transaction() called — Laravel sees existing open transaction
↓
2
SAVEPOINT trans2 created (not a new BEGIN)
↓
3
INSERT INTO claims — row appears, auto_increment ID = 247
↓
4
RELEASE SAVEPOINT — savepoint succeeds ✓
↓
5
API returns 201 Created { id: 247 } to client
↓
6
Outer phantom transaction rolls back — claim row vanishes. ID 247 is consumed forever.
Why this is hard to catch

Intermittent: Only triggers when a worker's previous request left a dirty connection — maybe 1 in 500 requests.

Invisible: Logging inside the transaction closure shows success because the savepoint really did release.

Time-dependent: If you query immediately after the response, the row might still be visible (the phantom transaction hasn't rolled back yet).

The Nuclear Option: DisconnectFromDatabases

Laravel Octane ships with a built-in listener for exactly this scenario:

phpconfig/octane.php
// In OperationTerminated listeners:
OperationTerminated::class => [
    FlushOnce::class,
    FlushTemporaryContainerInstances::class,
    \Laravel\Octane\Listeners\DisconnectFromDatabases::class,
],

This closes all database connections after every request. The next request opens a fresh connection. MySQL auto-rolls back any pending transaction on connection close. Problem solved.

The trade-off: You lose persistent connections — Octane's main performance advantage. Every request now pays the TCP handshake + authentication cost (~1–5ms within a Kubernetes cluster). For high-throughput APIs, this adds up.

The Balanced Fix: RequestReceived Guard

If you want to keep persistent connections for performance, you can add a guard that checks for dirty state before each request:

phpapp/Listeners/EnsureCleanDatabaseState.php
namespace App\Listeners;

use Illuminate\Support\Facades\DB;
use Illuminate\Support\Facades\Log;

class EnsureCleanDatabaseState
{
    public function handle($event): void
    {
        foreach (DB::getConnections() as $name => $connection) {
            // Kill phantom transactions
            if ($connection->transactionLevel() > 0) {
                Log::warning('Phantom transaction detected', [
                    'connection' => $name,
                    'level'      => $connection->transactionLevel(),
                    'worker_pid' => getmypid(),
                ]);

                $connection->rollBack(0);
            }

            // Verify connection is alive (handles K8s pod restarts)
            try {
                $connection->getPdo()->query('SELECT 1');
            } catch (\Throwable $e) {
                Log::warning('Stale connection, reconnecting', [
                    'connection' => $name,
                    'worker_pid' => getmypid(),
                ]);

                $connection->reconnect();
            }
        }
    }
}

Register it on RequestReceived:

phpconfig/octane.php
RequestReceived::class => [
    \App\Listeners\EnsureCleanDatabaseState::class,
    \App\Listeners\EnsureCleanRedisState::class,
],
What this catches

Open transactions from previous requests, dead connections after pod restarts or network blips, and transaction level mismatches.

What this doesn't catch

MySQL session variables (SET @var), temporary tables, advisory locks (GET_LOCK), and SET commands from previous requests. For full session cleanup without disconnecting, use RESET CONNECTION on MySQL 8.0+.

Defensive Code Patterns

Regardless of which infrastructure fix you choose, your application code should be resilient. The key principle: never trust data from inside a transaction closure after the closure returns. Always verify with a fresh query.

Pattern: INSERT Operations

phpClaimRepository — safe create
public function create(CreateClaimDto $dto): Claim
{
    // Transaction returns ONLY the ID — an integer, nothing else
    $claimId = DB::transaction(function () use ($dto) {
        $claim = $this->model->create($dto->toClaimArray());

        if (! $claim->exists) {
            throw new \RuntimeException('Failed to create claim');
        }

        // ... products, media, timeline inserts ...

        return $claim->id;
    });

    // AFTER transaction commits — verify the row actually exists
    $claim = $this->model->with(['user', 'products', 'media'])->find($claimId);

    if (! $claim) {
        throw new \RuntimeException(
            "Claim #{$claimId} not persisted. Worker: " . getmypid()
        );
    }

    return $claim;
}

Pattern: UPDATE Operations

phpClaimRepository — safe update
public function submitClaim(int $id, UpdateClaimDto $dto): ?Claim
{
    $claim = $this->model->find($id);

    if (! $claim) {
        return null;
    }

    // Don't return from inside the transaction
    DB::transaction(function () use ($claim, $dto) {
        if (! $claim->update($dto->toClaimArray())) {
            throw new \RuntimeException("Update failed for claim #{$claim->id}");
        }

        // ... timeline entries ...
    });

    // Fresh query AFTER commit — confirms persistence
    return $this->model->with(['user', 'products', 'media'])->find($id);
}
Why this matters

The fresh() or load() methods inside a transaction closure read from the same connection — they see uncommitted data. A find() after the transaction closure uses a committed read, confirming the row actually made it to disk.

Comparison: Which Strategy to Use

Strategy Latency Cost Coverage Safety
DisconnectFromDatabases ~1–5ms per request Everything — full reset Safest
RequestReceived guard ~0.1ms (SELECT 1 + check) Transactions + dead connections Safe
Guard + RESET CONNECTION ~0.2ms Transactions + session state + temp tables Near-full
Code-level verification only ~0ms (extra SELECT) Detects but doesn't prevent Detection only
No fix 0ms None Vulnerable

For Kubernetes environments where you need both performance and reliability, the RequestReceived guard combined with code-level verification gives you the best balance. You keep persistent connections, eliminate phantom transactions, and have a safety net in application code.

What About Redis?

Despite its name, DisconnectFromDatabases only targets connections through Laravel's DatabaseManager — that's MySQL, PostgreSQL, SQLite. Redis connections are managed by a separate RedisManager and are not affected by this listener.

Redis connections in Octane stay persistent across requests. This is generally fine — Redis handles thousands of concurrent connections easily and reconnection overhead is minimal. But in Kubernetes, the same stale-connection risk exists: if a Redis pod restarts, Swoole workers hold dead connections.

phpapp/Listeners/EnsureCleanRedisState.php
class EnsureCleanRedisState
{
    public function handle($event): void
    {
        try {
            $redis = $event->app->make('redis');

            foreach ($redis->connections() as $connection) {
                try {
                    $connection->ping();
                } catch (\Throwable) {
                    $connection->disconnect();
                }
            }
        } catch (\Throwable) {
            // No Redis connections yet
        }
    }
}

Final Production Config

phpconfig/octane.php — production ready
RequestReceived::class => [
    \App\Listeners\EnsureCleanDatabaseState::class,
    \App\Listeners\EnsureCleanRedisState::class,
],

OperationTerminated::class => [
    FlushOnce::class,
    FlushTemporaryContainerInstances::class,
    // DisconnectFromDatabases intentionally disabled for performance
    // CollectGarbage::class,
],

This configuration gives you persistent connections for low latency, phantom transaction protection at the start of every request, Redis health checks for Kubernetes resilience, and clear logging when dirty state is detected — giving you visibility into how often this actually happens in your environment.

Monitor before you optimize

After deploying the guard, watch your logs for "Phantom transaction detected" entries. If you see them frequently, investigate which endpoints are failing to commit or roll back properly — the guard is a safety net, not a substitute for fixing the root cause in your application code.

The ghost row problem is one of the most frustrating bugs in production Laravel Octane deployments. It passes every test, returns every correct response, and leaves no trace — until a user asks why their data disappeared. Understanding the mechanics of Swoole's persistent connections and MySQL's transaction nesting is the first step. Building the right guards — at the infrastructure level and in your application code — is what keeps your data alive.
